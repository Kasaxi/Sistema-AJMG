## Achados por Dimensão

### 1. Accessibility — 3/4

**OK:**
- ✓ Botões usam `<button>` ou `<div role="button" tabIndex={0}>` com keyboard handler
- ✓ Labels associadas a inputs via `htmlFor`/`id`
- ✓ Focus ring Cobalt 15% (`focus-visible:ring-[var(--brand-bright)]/15`) presente em formulários
- ✓ aria-labels em ícones de ação ("Marcar como concluída", "Mover pra cima")

**P1 — Focus visible inconsistente em alguns wrappers clicáveis**
- `src/components/agenda/task-card.tsx:51-65` adiciona focus-ring no `<div role="button">` ✓
- `src/components/agenda/views/semana-view.tsx:101-104` — coluna do dia tem `onClick` mas **sem** `role="button"` / `tabIndex` / focus handler. Teclado não navega
- `src/components/agenda/views/kanban-view.tsx` cards usam `<div draggable>` sem `role="button"` nem focus visível

**P2 — Hierarquia de headings**
- Várias páginas usam `<h1>` no Header + `<h3>` nos cards, pulando `<h2>`. Conformidade técnica WCAG OK, mas leitores de tela perdem nível semântico

**P3 — `aria-current` em sidebar**
- Item ativo no sidebar não tem `aria-current="page"`. Cosmético.

### 2. Performance — 3/4

**OK:**
- ✓ Sem layout thrash detectado em loops
- ✓ Lazy thumbnails nos anexos (`loading="lazy"` em `<img>`)
- ✓ Optimistic updates implementados (Kanban, check de tarefa)
- ✓ Skeletons enquanto carrega

**P2 — Refetch ao trocar de view em /agenda**
- Quando troca Kanban → Mês, o `useEffect` dispara fetch novo. Pra a maioria dos casos, os itens já estavam em memória da view anterior. Pequeno desperdício.

**P2 — Re-renders no TaskModal**
- Modal recria categorias/pessoas a cada open. Memoizar com useMemo poderia evitar.

**P3 — Carga inicial de anexos**
- `getAnexoSignedUrl` é chamado por anexo no mount do AnexosGallery. Pra 10 anexos = 10 round-trips. Poderia ser batch.

### 3. Responsive Design — 3/4

**OK:**
- ✓ Sidebar vira drawer no mobile com backdrop
- ✓ Headers sticky com `backdrop-blur`
- ✓ Touch target dos cards e botões principais ≥ 44px
- ✓ Modal `sm:max-w-2xl` + `max-h-[92vh] overflow-y-auto`

**P1 — Kanban drag-and-drop no mobile**
- HTML5 native drag não funciona bem em touchscreens. Solução: detectar touch e oferecer mover entre status via **long-press → bottom sheet com opções**. Omar usando no celular vai sofrer

**P2 — Select compacto em mobile**
- Selects pequenos (h-9, h-8) em filtros/configurações têm touch target borderline em iOS. DESIGN.md pede 44px mínimo em mobile

**P2 — Calendário Mês em telas pequenas**
- 7 colunas × 6 linhas fica apertado em < 380px. Cell rola horizontal? Texto trunca? Faltou testar

**P3 — Header de Vendas/Clientes tem muitos filtros**
- Em mobile, filtros viram coluna gigante. Considerar drawer/bottom-sheet de filtros

### 4. Theming — 3/4

**OK:**
- ✓ 95% das cores via `var(--brand)`, `var(--ink)`, `var(--line)`, etc
- ✓ Sem gradients (0 ocorrências)
- ✓ Sem `#000` ou `#fff` puro fora de SVG
- ✓ Sem `bg-gradient-*` em uso

**P1 — StatusBadge usa 6 paletas Tailwind paralelas**
- `src/components/vendas/status-badge.tsx:35-45` usa `bg-slate-100`, `bg-amber-50`, `bg-orange-50`, `bg-rose-50`, `bg-violet-50` direto. DESIGN.md prevê só: brand-tint, hairline, cool-paper, e os warn-pointuais (rose, amber, emerald). **Slate, orange e violet são extras**. Considerar reduzir a paleta de status pra alinhar com o sistema.
- *Risco baixo* — o componente funciona; só não escala junto com a paleta nomeada

**P2 — Cores hardcoded em comentários/dashboard**
- Alguns hex como `#06B6D4`, `#14B8A6`, `#7C3AED`, `#EC4899` aparecem (provavelmente em gráficos do dashboard de vendas). DESIGN.md não cobre cor pra data viz ainda — quando começar Compras/Financeiro precisará definir esse mini-sistema.

### 5. Anti-Patterns — 3/4

**OK:**
- ✓ Sem emojis em headings (0 ocorrências de 🚀📊✅⚡🎉)
- ✓ Sem gradient text
- ✓ Sem side-stripe borders (border-left ≥ 3px)
- ✓ Sem cards aninhados (NestingCardCheck OK)
- ✓ Microcopy direto: "A fazer", "Fazendo", "Feita", "Atrasada"
- ✓ Sidebar escuro respeita o "Site Night"
- ✓ Cards têm hover-lift, não sombra em repouso

**P1 — Violação possível da "Regra dos 10% Cobalt"**
- `[var(--brand-bright)]` aparece em ~30+ lugares: tabs ativas, sidebar item ativo, focus rings, dots de status, ícones. Não medi a % por tela, mas vale auditar visualmente em cada surface — pode estar passando dos 10% no Header + Sidebar simultaneamente
- *Mitigação:* manter ou aceitar como "regra mais relaxada que aspirational"

**P2 — Múltiplos paletas ortogonais em StatusBadge**
- Mesma issue do tema #4. Repetida aqui porque é também violação da paleta DESIGN.md.

**P3 — Comentários em código com regras de design**
- Comentários como "VENDA_FECHADA usa azul-brand (regra 'nunca verde')" mostram que a equipe lembrou da regra. Bom. Mas isso é doc de código — deveria estar em DESIGN.md (e agora está).

## Pontos não auditados (limitações)

Esta foi uma **varredura por amostragem**. Não cobre:
- Contraste medido por pixel em cada combinação (precisaria de ferramenta a11y como axe)
- Lighthouse performance score real
- Visual regression em todos os breakpoints
- Teste com leitores de tela (NVDA/VoiceOver)
- Acesso real com touchscreen
